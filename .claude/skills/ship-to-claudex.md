# Ship to Claudex

Canonical capability: claudex.skill.ship-to-claudex

Trigger this skill when KP or Rory says a new topic, loop, hardcoded rule, reusable lesson, skill, control, or system advancement should become part of Claudex. Also trigger it before session close when current authorized work produced a verified reusable Claudex capability.

## Contract

Read `OPS/CLAUDEX_CAPTURE_AND_SHIP.md` in full.

1. Apply the capture gate. Convert the durable behavior into a narrow contract. Never ship raw conversation.
2. Inspect current Claudex skills, memory, contracts, backlog, bridge, questions, directives, and receipts. Extend the existing brain instead of creating a duplicate.
3. Route the advancement to one primary authority file. Add discovery pointers only for consumers that need them.
4. Before creating files, resolve every listed artifact with `npm run content:route` and record its precise name, artifact type, product or system, canonical path, source authority, consumers, and acceptance evidence.
5. Preserve product repository boundaries. Never copy secrets, credentials, transcript dumps, private data, or product implementation into Claudex.
6. Put uncertain ideas in `CLARIFY`, `BLOCKED`, or `PARKED`. Never hardcode inference as verified truth.
7. Add deterministic validation when executable behavior or structured state changes.
8. Use the command layer for bridge and receipt writes. Run the narrow tests, `npm run check`, the secret scan, and `git diff --check`.
9. Review the exact scope, publish it to the canonical default branch, and read back the resulting remote commit.

Inspect actual repository scripts before naming a publication command. Never invent a bridge or publish command that is not present.

Do not claim shipped until the remote commit and its file set are verified. Report `PARTIAL` or `BLOCKED` when evidence, authority, validation, or publication is incomplete.
