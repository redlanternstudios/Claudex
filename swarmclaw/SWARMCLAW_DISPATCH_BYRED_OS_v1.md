## Plan

### 1. Shared source order

The order of truth is:
1. Claudex bridge and receipts
2. By Red OS task record
3. GitHub repo history
4. Supabase data
5. Obsidian mirror
6. Drive mirror
7. Chat history only as temporary context

### 2. Work flow

Use this sequence everywhere:
1. Plan
2. Build
3. Test
4. Ship
5. Record
6. Mirror
7. Review

### 3. Loop design

Every project needs these loops:
1. Intake loop for new ideas and requests
2. Build loop for active work
3. Test loop for proof and receipts
4. Ship loop for release
5. Memory loop for RAG and retrieval
6. Review loop for cleanup and next steps

### 4. Triggers and hooks

When any of these happen, the system must react:
1. New chat starts
2. New task starts
3. New receipt is written
4. Build state changes
5. A model or provider changes
6. A repo changes
7. A database schema changes

Each trigger should update the visible record and the mirror record.

### 5. RAG layers

Use layered retrieval so each chat starts with the right context.

Layer order:
1. Current bridge state
2. Current project task record
3. Current receipt trail
4. Project docs
5. Mirror docs in Obsidian and Drive
6. Older chat memory only if needed

### 6. Model change policy

If a new model arrives that is better, cheaper, or more efficient:
1. Treat it as a candidate, not a default.
2. Compare cost, quality, speed, and stability.
3. Run a small real test lane.
4. Record the result in a receipt.
5. Update routing only after evidence, not hype.

If the new model is worse for a lane, keep the current route.

### 7. Project record standard

Every project record should show:
1. What it is
2. Who owns it
3. What stage it is in
4. What is blocking it
5. What the next action is
6. Where the proof lives
7. Where the mirror lives

### 8. By Red OS role

By Red OS is the project ledger.

It should hold:
1. Tasks
2. Status
3. Receipts
4. Source registry
5. Mirror pointers
6. Clear ownership

### 9. Claudex role

Claudex is the shared control plane.

It should hold:
1. Bridge state
2. Receipts
3. Directives
4. Product focus
5. Sync color
6. Handoff truth

### 10. SwarmClaw role

SwarmClaw is the build engine.

It should:
1. Read Claudex first
2. Build only what is locked
3. Use local models first for routine work
4. Use paid providers only when a lane is named and a receipt says why
5. Write back evidence when work is done

## Missing Pieces To Finish

1. One readable lane map for every active project.
2. One shared task naming rule so Ro and Keymon can scan it fast.
3. One receipt rule that ties each change to proof.
4. One mirror rule for Obsidian and Drive.
5. One model routing rule that handles future better models without churn.

Do not let a mirror become the only truth.
Do not let a model switch happen without evidence.
