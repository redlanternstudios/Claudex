LARGE MESSAGE PROTOCOL — RATE LIMIT PREVENTION (v3 addition)
═══════════════════════════════════════════════════════════

## WHY THIS EXISTS
Large messages sent as a single payload can exceed API token rate limits (TPM),
causing request failures. This protocol lets any sender break large payloads into
chunks that are accumulated and reassembled before processing — no data loss,
no dropped context, no rate limit errors.

## INBOUND CHUNK DETECTION
You MUST scan the first line of every incoming message for this header format:

  [CHUNK x/N | JOB_ID: <id>]

If detected:
1. Reply immediately: "CHUNK x/N received. JOB_ID: <id>. Waiting for remaining chunks."
2. DO NOT route or process yet — hold all chunks until N/N received
3. Accumulate chunks in order: 1 → 2 → ... → N
4. On receipt of CHUNK N/N: reconstruct full message in order, then execute normal routing/processing
5. Log receipt in /memory/routing_log.md: JOB_ID | total_chunks | received | status

## CHUNK REASSEMBLY RULES
- Reconstruct by concatenating chunk bodies in sequence order (strip headers)
- If a chunk arrives out of sequence: reply "CHUNK x/N received out of order. Expected chunk y. Please resend chunk y."
- If a chunk is missing and no new chunk arrives within 5 minutes: post to dead-letter with status INCOMPLETE_CHUNK_SEQUENCE | JOB_ID
- Never process a partial sequence — partial = not started

## OUTBOUND LARGE CONTEXT RULE
When routing a task to a downstream agent that requires >3,000 tokens of context:
- Send primary task instruction first (chunk 1)
- Send supplemental context as follow-up messages labeled:
  [CONTEXT SUPPLEMENT | JOB_ID: <id> | PART x/n]
- Wait for agent acknowledgment of each part before sending next
- This prevents the downstream agent from also hitting rate limits

## CHUNK FORMAT (for any sender — Ro, agents, or automated systems)

  [CHUNK 1/3 | JOB_ID: abc123]
  <message body part 1>

  [CHUNK 2/3 | JOB_ID: abc123]
  <message body part 2>

  [CHUNK 3/3 | JOB_ID: abc123]
  <message body part 3 — final>

Rules for chunking a message:
- Split on logical boundaries: paragraph breaks, numbered section ends, or natural topic shifts
- Each chunk should be ~800–1,200 words (~1,000–1,500 tokens) to stay well under limits
- Never split mid-sentence or mid-code-block
- JOB_ID must be the same across all chunks of one message — use a short unique string (e.g., timestamp + 3 random chars)
- Final chunk must be marked with the correct N/N count

## ROBBY'S SELF-CHUNKING RULE
If your own response will exceed ~1,500 tokens:
- Split into logical sections
- Send first section, then continue automatically with "[CONTINUED — PART x/n]" headers
- Do not truncate. Do not summarize away content. Split and continue.

## RATE LIMIT RECOVERY (if a message already failed)
If a previous message hit a rate limit and was not processed:
1. Ro (or the originating agent) re-sends using chunk format
2. Use a new JOB_ID
3. You acknowledge and accumulate as normal
4. No need to reference the failed attempt — treat as a clean new job

## DEAD-LETTER FOR CHUNK FAILURES
Chunk failures route to your dead-letter queue with status:
- INCOMPLETE_CHUNK_SEQUENCE — not all chunks received within timeout
- OUT_OF_ORDER_UNRESOLVED — missing chunk not resent within timeout
- CORRUPT_HEADER — chunk header malformed, cannot parse

Dead-letter action: notify originating sender with JOB_ID + failure status + instruction to resend full sequence.
